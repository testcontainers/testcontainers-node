# 새 모듈 추가 기회

## 개요

testcontainers-node는 41개 모듈을 보유하고 있으며, 새 모듈 PR을 적극 수용. Oracle Free 모듈(PR #1242, `collinmurd`)이 17일 만에 머지된 선례.

## 검증된 패턴

기존 모듈을 템플릿으로 사용:
1. `packages/modules/` 아래 새 디렉토리 생성
2. `package.json`, `src/`, `__tests__/` 구조
3. 컨테이너 설정 + Wait Strategy
4. 통합 테스트
5. README 문서

## 후보 서비스 (아직 모듈 없음)

| 서비스 | 유형 | Node 생태계 인기 | 난이도 | 이유 |
|--------|------|----------------|--------|------|
| **Meilisearch** | 검색 엔진 | 높음 | 낮음 | Node SDK 인기, 간단한 HTTP API |
| **SurrealDB** | 멀티모델 DB | 중상 | 낮음-중간 | 차세대 DB, 커뮤니티 성장 중 |
| **DragonflyDB** | Redis 호환 인메모리 | 중간 | 낮음 | Redis 모듈 패턴 그대로 적용 가능 |
| **Turso/libSQL** | 분산 SQLite | 중상 | 중간 | Edge computing 트렌드 |
| **Temporal** | 워크플로 엔진 | 중간 | 중간 | 마이크로서비스 테스팅에 필수적 |

## 추천: 실제 사용하는 서비스

포트폴리오에서 "이 서비스를 실제 프로젝트에서 사용하면서 테스트 모듈이 없어서 직접 만들었다"는 스토리가 가장 강력.

## 머지 확률

**중상** — Oracle Free 모듈(PR #1242) 17일 머지 선례가 있지만, 새 모듈 수용은 개별 심사. 서비스의 인지도, 장기 유지보수 부담, 테스트 안정성에 따라 달라짐. 한 건의 선례만으로 일반화하기 어려움.

수용 가능성을 높이려면:
- 충분한 테스트 커버리지
- 문서 포함
- 기존 모듈과 일관된 API 스타일
- 실제 사용 수요가 있는 서비스 선택

## 포트폴리오 가치

**중상** — 재사용 가능한 오픈소스 모듈 저자. 다만 코어 기여보다는 시그널이 약할 수 있음.

## 실행 계획

1. 모듈로 만들 서비스 선정 (실제 사용 중인 것 우선)
2. 기존 모듈 (예: `packages/modules/redis`) 구조 분석
3. 컨테이너 설정 + Wait Strategy 구현
4. 통합 테스트 작성
5. README 문서 작성
6. PR 제출
